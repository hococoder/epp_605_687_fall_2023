Wrapping a UIKit view in SwiftUI

As mentioned previously, not all UIKit views have SwiftUI counterparts. So, Apple added some support into SwiftUI to wrap UIKit views (and also added support into UIKit to support the wrapping of SwiftUI views, in case you were still using UIKit but had a great SwiftUI view to use). Take a look at this class to present a mail composition view in SwiftUI. The commented numbers in the code correspond to the explanations below.

import SwiftUI
import MessageUI

//1
struct MailView: UIViewControllerRepresentable {

    @Binding var isShowing: Bool
    @Binding var result: Result<MFMailComposeResult, Error>?

    //2 - The Coordinator helps bridge between SwiftUI and the delegate for the UIKit object
    class Coordinator: NSObject, MFMailComposeViewControllerDelegate {

        @Binding var isShowing: Bool
        @Binding var result: Result<MFMailComposeResult, Error>?

        init(isShowing: Binding<Bool>,
             result: Binding<Result<MFMailComposeResult, Error>?>) {
            _isShowing = isShowing
            _result = result
        }

        func mailComposeController(_ controller: MFMailComposeViewController,
                                   didFinishWith result: MFMailComposeResult,
                                   error: Error?) {
            defer {
                isShowing = false
            }
            guard error == nil else {
                self.result = .failure(error!)
                return
            }
            self.result = .success(result)
        }
    }

    //3 - returns the coordinator
    func makeCoordinator() -> Coordinator {
        return Coordinator(isShowing: $isShowing,
                           result: $result)
    }

    //4 - Handles the creation of the controller, and assigns the context to it
    func makeUIViewController(context: UIViewControllerRepresentableContext<MailView>) -> MFMailComposeViewController {
        let vc = MFMailComposeViewController()
        vc.mailComposeDelegate = context.coordinator
        return vc
    }

    //5 - We're not updating here, so this method stays blank
    func updateUIViewController(_ uiViewController: MFMailComposeViewController,
                                context: UIViewControllerRepresentableContext<MailView>) {

    }
}

Here we are:

  1. Stating that the view adopts the UIViewControllerRepresentable protocol. This protocol builds on the View protocol, which we've used to make our views so far in SwiftUI. We're not defining a body property, since we'll use the view provided by the UIView from UIKit.
  2. Defining a coordinator class object. This is the bridge between SwiftUI and UIKit that allows SwiftUI to delegate responsibility for interactions back to the Controller, which knows how to interface with the view from UIKit (and vice versa). This coordinator must adopt the delegate protocol for the UIKit view, so here it adopts MFMailComposeViewControllerDelegate. We're also inheriting the class from NSObject, which is the base for almost all objects in UIKit.
  3. Defining a method to initialize the Coordinator defined above - SwiftUI automatically calls this method if we implement it.
  4. Implementing the first of the UIViewControllerRepresentable protocol methods - makeUIViewController. This method takes in a UIViewControllerRepresentableContext, which is templated to our SwiftUI view, and returns the controller from UIKit that we want to present to the user.
  5. Implementing the second of the UIViewControllerRepresentable protocol methods - updateUIViewController. We won't need to do anything with this method, so we'll leave the body blank.

Note here that we're using bindings to keep track of the data passed into not only the MailView, but the Coordinator as well. Changes here will get percolated back up to where we call MailView (which will be from the StudentView). One additional note about this view - it will only work on device unless you have a mail account setup on your simulator.
